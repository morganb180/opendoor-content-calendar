## 1. What this product is

A social content calendar for Opendoor IG/FB posts. One person (Morgan) produces the assets;
a small team views the calendar, previews assets, and leaves comments. Posting cadence is
~3×/week plus event specials (holidays, World Cup, Wimbledon…).

Assets are produced in two work modes, and the calendar must serve both:

- **Claude mode (local generation):** images/videos are generated by local AI pipelines in
  `~/opendoor/projects/*`, finished files land in `~/opendoor/finals/<spot>/`. A Claude Code
  session finishes an asset and should be able to register it on the calendar with **one command**.
- **Canva mode (preview/deploy):** most static posts also live as Canva designs (IDs like
  `DAHxxxxxxxx`). Canva is where final tweaks, exports, and eventually publishing happen. The
  calendar links each post to its Canva design and should be able to pull a design's
  title/thumbnail rather than requiring hand-entry.

The calendar itself is the **team visibility layer**: month grid, schedule list, unscheduled
inventory, full-asset lightbox previews (carousels + video), holiday/event markers, comments.

## 2. Current architecture (as-is)

Static, no build step, all client-side:

```
index.html      markup (views + 3 modals/lightbox)
styles.css      all styles, responsive
app.js          820 lines: seeded content, state, rendering, DnD, comments,
                GitHub sync, Canva import parser, Canva Autofill push, password gate
previews/       ~33 MB of preview PNG/JPGs (+ previews/slides/ for carousels)
scripts/smoke-test.js   node script: parses app.js, greps for required markup/functions,
                        verifies referenced preview assets exist
```

State model today:

- Seed content is **hardcoded in `app.js`** (`DEFAULTS` = scheduled posts, `INVENTORY` = ready
  assets, `MARKERS` = holidays).
- Runtime state lives in **localStorage** (several keys, payload `version:6`).
- Cross-machine sync = **`calendar-data.json` committed to the repo via the GitHub Contents API**
  from the browser ("Sync ↑" asks for a fine-grained PAT, stored in localStorage). "Sync ↓" /
  first-run auto-pull fetches that file.
- Password gate: SHA-256 hash compare in JS, deterrence only (site + source are public).
- Canva: "Import from Canva" parses pasted URLs/batch rows into drafts (client-only, works).
  "Push to Canva Autofill" calls the Canva API directly from the browser (broken — see F4).

### Known constraints (do not violate)

- **Keep GitHub Pages hosting and the existing URL.** No paid infra. A free Cloudflare Worker is
  the only server-side component permitted, and only in Phase 3 if needed.
- **Keep the no-build-step property of the deployed site.** Node is allowed for CLI tooling,
  tests, and CI, but `index.html` must keep working by opening the deployed page — no bundler.
- **Never use fal.ai** for anything (hard rule from Morgan), including as a file-to-URL bridge.
- **Opendoor logo:** always the official `opendoor_wordmark.svg`; never redraw it. (Applies if
  you touch branding; the current inline house icon in the header is fine to keep.)
- Keep the password gate (same UX, deterrence-only is accepted and documented).
- **Do not lose user data:** existing localStorage state (v6 payload) must migrate cleanly.
- Preview images live in the repo; videos do NOT get committed to this repo (they're hosted
  externally — see §6.3).

## 3. Audit findings (fix list)

Severity: 🔴 broken/risky now · 🟡 will bite soon · ⚪ cleanup.

- **F1 🔴 Sync has never run.** `calendar-data.json` 404s on Pages and raw.githubusercontent.
  The entire team-sync + comments-sync story is dormant; fresh machines silently fall back to
  seeds (`pullFromRepo` swallows the error). Once Phase 1 lands, the committed data file always
  exists, which retires this class of failure.
- **F2 🔴 Three competing sources of truth.** Content lives in (a) `DEFAULTS`/`INVENTORY` in
  app.js, (b) each browser's localStorage, (c) the synced JSON. The reconciliation patch in
  `load()` only merges (a)→(b) on load; `applyPayload()` (sync-pull/import) wholesale replaces
  posts, so a pull can permanently hide posts that were added to code after the last push.
  **Root-cause fix is Phase 1 (data-first refactor), not more merge patches.**
- **F3 🔴 Hardcoded clock.** `const TODAY="2026-06-30"` (app.js:8) and `goToday()` hardcodes
  July 2026. Today-highlight and the Today button are frozen. Compute from `new Date()`.
- **F4 🔴 Canva Autofill push is broken as built** (`pushCaptionToCanva`):
  - Wrong base path: uses `https://api.canva.com/v1/autofills`; Canva Connect is
    `https://api.canva.com/rest/v1/autofills`.
  - Autofill create is an **async job** — the code fire-and-forgets and reports success without
    polling `GET /rest/v1/autofills/{jobId}`.
  - Token model mismatch: Connect uses short-lived OAuth 2.0 (PKCE) access tokens, not a
    long-lived paste-once token; and browser calls from an arbitrary Pages origin will hit CORS.
  - The Autofill API requires **Canva Enterprise**; account is Pro. Feature is likely
    unavailable regardless of code.
  **Decision: remove the in-browser Autofill push in Phase 0** (keep "Copy caption", which
  works). Canva integration returns properly in Phase 2 via local tooling (§6.2).
- **F5 🔴 Attribute injection / token theft surface.** Many innerHTML builds escape text nodes
  with `esc()` but interpolate **attributes unescaped**: `img src="'+(p.img||"")+'"` (grid chips,
  cards, lightbox), `href="'+canvaURL(...)+'"`, `poster=`, `video src=`. A crafted value in an
  imported JSON / synced payload / Canva batch row (e.g. `x" onerror="...`) executes JS — and a
  GitHub PAT sits in localStorage on this origin. Escape every interpolated attribute (or build
  DOM nodes and set properties), and validate/whitelist URL schemes (`https:`, relative,
  `file:` for the documented local-video case).
- **F6 🟡 Data-URI uploads will blow the storage quota.** `uploadPrev` base64s any image into
  the post record → localStorage (~5 MB) and the synced JSON inflate fast. Phase 0: cap upload
  size (~300 KB) with a clear error pointing at the proper flow; Phase 2: `calctl` puts real
  files in `previews/` instead.
- **F7 🟡 Sync is last-write-wins across the whole payload.** The conflict guard only compares
  timestamps and `confirm()`s. Two people editing (or one person on two machines) clobber each
  other's **comments** especially. Phase 3 does per-entity merge; until then the UI should say
  "pull before you edit" and auto-pull on load (not just first run).
- **F8 🟡 Global-not-per-post Canva caption field.** The edit form shows "Canva caption field"
  per post, but it saves to a single global localStorage key on every keystroke. Confusing;
  remove with F4.
- **F9 ⚪ 33 MB of unoptimized previews** and growing; several are full-resolution exports.
  Phase 2 adds automated downscale/compression (target ≤200 KB grid thumbs; keep full-res only
  inside the lightbox where needed).
- **F10 ⚪ Monolith app.js (820 lines).** Acceptable for the no-build constraint, but split into
  ES modules (`<script type="module">` works fine on Pages) during Phase 1: `data.js`,
  `render.js`, `sync.js`, `canva.js`, `gate.js`, `main.js`.
- **F11 ⚪ Smoke test is string-grep only.** Keep it (it catches real regressions cheaply), but
  Phase 1 adds schema validation of the data file + a DOM smoke test (see §8).
- **F12 ⚪ `commentKey` produces `inv-inv-<slug>` ids** (double prefix). Harmless but confusing;
  normalize during the Phase 1 migration (migrate existing comment keys).

## 4. Target architecture

**Principle: the repo is the database.** Content lives in committed JSON; the site is a pure
renderer; a CLI is the write path for both work modes; the browser UI remains a convenient
editor that writes through the same sync file. Everything else (CI validation, image
optimization, Canva enrichment) hangs off that spine.

```
                    ┌───────────────────────────── repo (source of truth) ─┐
 Claude mode ──────▶│ data/calendar.json   ← posts, inventory, markers,    │
 (calctl add/       │                        comments, config              │
  deploy)           │ previews/…           ← optimized images              │──▶ GitHub Pages
                    │ index.html + js/ css │                               │    (viewer + light editor)
 Canva mode ───────▶│ tools/calctl         ← Node CLI (no deps)            │
 (calctl canva      └───────────────────────────────────────────────────────┘
  pull/import)            ▲                        │
                          │ GitHub Actions: validate schema, check assets,
 Browser "Sync ↑" ────────┘ optimize images, (weekly) link-check Canva URLs
 (unchanged mechanism, now writing data/calendar.json)
```

### 4.1 Data model (`data/calendar.json`, schema `version: 7`)

```jsonc
{
  "version": 7,
  "updatedAt": "2026-07-02T21:00:00Z",     // set by whoever writes the file
  "config": {
    "cadence": "Mon/Wed/Fri + event specials",
    "timezone": "America/Los_Angeles",
    "themes": {                             // moved out of code
      "seasonal": {"color": "#ff7a45", "label": "Seasonal"},
      "worldcup": {"color": "#16a34a", "label": "World Cup"},
      "meme":     {"color": "#7c3aed", "label": "Meme"},
      "proof":    {"color": "#0ea5e9", "label": "Proof"},
      "brand":    {"color": "#1f80ff", "label": "Brand"}
    }
  },
  "markers": { "2026-07-04": {"label": "🇺🇸 Independence Day", "type": "hol"} },
  "posts": [
    {
      "id": "p-abc123",
      "status": "scheduled",               // draft | ready | scheduled | posted
      "date": "2026-07-04",
      "title": "Fireworks Wordmark",
      "theme": "seasonal",
      "fmt": "Video · 9:16",
      "holiday": "July 4th",
      "caption": "internal notes…",
      "socialCaption": "the live IG/FB caption",
      "canva": "DAHOFu7Evxs",              // design id or full URL
      "media": {
        "img": "previews/fireworks.png",   // grid/list thumbnail
        "slides": ["previews/slides/…"],   // optional carousel
        "video": "https://…mp4"            // optional; https URL preferred
      },
      "postedUrl": "",                     // filled once live (status: posted)
      "updatedAt": "2026-07-01T18:00:00Z"
    }
  ],
  "inventory": [ /* same shape minus date/holiday/status */ ],
  "comments": { "p-abc123": [ {"id": "c-…", "author": "Morgan", "text": "…", "ts": 1730000000000} ] }
}
```

Rules:

- `app.js` contains **zero content**. `DEFAULTS`, `INVENTORY`, `MARKERS`, `THEME` all move into
  the data file (one-time extraction script; also migrate legacy v6 localStorage payloads and
  legacy comment keys per F12 on first load).
- Every writer (CLI, browser save, sync push) bumps the entity's `updatedAt` and the top-level
  `updatedAt`.
- IDs are stable and permanent. Inventory→schedule promotion keeps the id (moves the record,
  preserving comments).
- A JSON Schema for this file lives at `data/calendar.schema.json`; `calctl validate` and CI
  both enforce it (see §8).
- Back-compat: the browser still mirrors state to localStorage for offline edits, but on load
  it fetches `data/calendar.json` and reconciles by `updatedAt` per entity (newer wins),
  replacing today's replace-all `applyPayload`.

### 4.2 The viewer/editor (site)

Feature-freeze the UI at parity, minus removals, plus small additions:

- Keep: month grid, list, inventory, DnD, lightbox w/ carousels & video, comments, add/edit
  modals, export/import JSON, password gate, mobile layout.
- Remove (Phase 0): "Push to Canva Autofill" button + per-post caption-field input (F4/F8).
  Keep "Copy caption".
- Add: `status` chip (draft/ready/scheduled/posted) on cards + a status select in the edit
  modal; "posted" posts render dimmed with a link if `postedUrl` set.
- Add: a subtle "data as of <updatedAt> · Sync ↓ for latest" indicator in the toolbar; auto-pull
  on every load when remote `updatedAt` is newer than local (silent, before first render).
- Sync ↑ keeps the current PAT mechanism but writes `data/calendar.json` (not the root file) and
  is now labeled "Publish changes".

### 4.3 `calctl` — the CLI write path (Node ≥18, zero npm dependencies)

Lives at `tools/calctl` (executable, `#!/usr/bin/env node`, plus `tools/lib/*.js`). Operates on
the repo checkout it lives in. Commands:

```
calctl add        --title "…" --theme meme --fmt "Static · 3:4"
                  [--date 2026-08-05 | --inventory]
                  [--img /path/to/preview.png]        # copied into previews/, auto-optimized
                  [--slides dir-or-glob]              # copied into previews/slides/<slug>-N.png
                  [--video URL | /path/file.mp4]      # https URL stored as-is; local path →
                                                      #   prints hosting instructions (§6.3) and
                                                      #   stores the URL you give back, or --allow-local
                  [--canva DAHxxx|url] [--caption "…"] [--social-caption "…"] [--holiday "…"]
calctl schedule   <id|title-substring> <date>         # inventory → calendar (id preserved)
calctl move       <id> <new-date>
calctl set        <id> key=value […]                  # status=posted postedUrl=… etc.
calctl rm         <id>
calctl list       [--month 2026-07] [--inventory] [--status ready] [--json]
calctl canva pull <id|--all>                          # Phase 2: fetch title/thumbnail via Connect API
calctl import-canva "<url>[, date, theme, title, caption]" …   # same parser as the UI, headless
calctl validate                                       # schema + referenced assets exist + dates sane
                                                      # + duplicate canva ids + image budget (≤300KB warn)
calctl deploy     [-m "message"]                      # validate → git add data/ previews/ → commit → push
                                                      # prints the Pages URL; refuses if validate fails
```

Design requirements:

- Zero npm dependencies (stdlib + built-in `fetch`). Image downscale via macOS `sips` when
  available, else warn (CI re-optimizes anyway).
- Video thumbnails: if `--video` is a local file and no `--img` given, grab a poster frame via
  `ffmpeg` when available.
- Idempotent and agent-friendly: stable exit codes, `--json` output for list/validate, clear
  one-line errors. This CLI is how Claude sessions will register finished assets, so every
  command must be safely re-runnable.
- `calctl deploy` never force-pushes; on non-fast-forward it instructs to pull/rebase.

### Phase 1 — Data-first refactor (the big one)

1. Extraction script: `DEFAULTS`/`INVENTORY`/`MARKERS`/`THEME` → `data/calendar.json` (v7),
   preserving current seed ids so existing comments/localStorage map over; legacy comment-key

2. `app.js` → ES modules (F10); render from fetched data; per-entity `updatedAt` reconcile
   replaces `applyPayload` replace-all (F2, F7 partial).
3. Auto-pull on every load when remote is newer; "data as of" indicator; status field + chips.
4. `data/calendar.schema.json` + `calctl validate` (validate may land here even though the rest

**Checkpoint:** parity review on live site (grid/list/inventory/lightbox/comments/DnD/mobile),
migration test from a captured v6 localStorage payload, schema review.

### Phase 2 — Ingestion & Canva enrichment

1. Full `calctl` per §4.3 (add/schedule/move/set/rm/list/import-canva/deploy).
2. Image pipeline: auto-optimize on `calctl add` + a CI job that fails if any preview >300 KB
   (with an auto-fix script), one-time crunch of the existing 33 MB (F9).
3. `calctl canva pull`: OAuth PKCE flow run **locally** (localhost redirect), token cached in
   `~/.config/opendoor-calendar/`; fetches design title + thumbnail via Connect API
   (`GET /rest/v1/designs/{id}`), downloads thumbnail into `previews/`. Degrade gracefully if
   scopes/tier block an endpoint — always leave the record valid.
4. Video hosting convention (§6.3) documented + `calctl` local-video handoff implemented.
5. `docs/WORKFLOW.md` + `CLAUDE.md` (§4.4).

**Checkpoint:** end-to-end demo: register a fake asset via CLI → deployed on Pages in one
command; Canva pull on 3 real design ids; image budget green.

### Phase 3 — Team workflow hardening (scope-check with Morgan before starting)

1. Merge-safe sync: per-entity `updatedAt` merge on push too (pull-merge-push in one action);
   comments merge by comment id (union) so two commenters never clobber each other (F7 full).
2. Roles: viewers comment without any token — **option A (preferred, zero infra):** comments
   Sync ↑ via a fine-grained PAT restricted to `data/` isn't safe to share, so instead route
   comment-writes through a tiny Cloudflare Worker holding the token (only endpoint:
   append-comment). **Option B:** keep comments Morgan-mediated (team uses UI, Morgan syncs).
   Decide with Morgan at phase start.
3. Activity log (`data/activity.jsonl`, appended by calctl/sync; simple "recent changes" panel).
4. Weekly CI link-check: Canva URLs still resolve, video URLs 200, flags via a GitHub issue.

### Phase 4 — Nice-to-haves (backlog, pick by demand)

Quarter view · filter by theme/status · IG-style feed preview mode · CSV/Sheets export ·
"posted performance" fields · Slack/email weekly digest via Actions cron · publish-reminder

## 6. Integration specifics

### 6.1 GitHub sync (browser)

Keep the current mechanism (Contents API PUT with fine-grained PAT, localStorage-stored) —
it's the right zero-infra choice — but: target `data/calendar.json`, pull-merge-push
(Phase 3.1), never store the PAT before validating it with a test GET, and clear it on 401/403
(already does).

### 6.2 Canva

- v1 (Phase 0–1): Canva = links out (`canva.com/design/<id>/view`) + the existing client-side
  URL/batch import parser. This works today; don't break it.
- v2 (Phase 2): local OAuth via `calctl canva pull` for titles/thumbnails as in §4.3. **Do not**
  reintroduce browser-side Canva API calls; the token/CORS/tier model doesn't fit a static page.
- Explicitly out: Autofill API (Enterprise-gated), auto-publishing to IG/FB.

### 6.3 Video hosting

Videos are not committed to this repo. Convention: upload finished mp4s to the existing
`github.com/morganb180/canva-asset-host` repo (raw URLs already power one post) or another
https host Morgan names; store the https URL in `media.video`. `calctl add --video <local>`
prints exactly this instruction (and with `--allow-local` stores the `~/…` path for
Morgan-machine-only playback, current behavior). One Phase 2 task: sweep the ~20 inventory
videos currently pointing at `~/opendoor/finals/...` and either upload+relink or tag them
`localOnly: true` so the UI can label the button honestly ("plays on Morgan's machine only").

## 7. Non-goals

- No framework migration (React/Vite/etc.), no bundler, no npm runtime deps for the site.
- No real auth (documented deterrence gate stays), no user accounts.
- No direct social publishing; the calendar schedules and previews, humans post.
- No CMS/database service; the repo is the database.

## 8. Testing & CI

- Keep `scripts/smoke-test.js`; extend per phase (it's cheap and has caught regressions).
- Add `scripts/validate-data.js` (used by calctl + CI): JSON-Schema check, asset existence,
  ISO dates, theme keys valid, no duplicate ids, image size budget.
- Add a headless DOM smoke test (node + jsdom is acceptable as a devDependency only, or use a
  Playwright-in-CI job): gate unlock → grid renders N chips → open lightbox → add comment →
  export JSON round-trips through import.
- GitHub Actions: `test.yml` on push/PR (smoke + validate + DOM test); `linkcheck.yml` weekly.
- Every phase PR must show: tests green, live-site manual checklist filled into the PR body.
